Post‑MVP: Режимы решения тренера
Контекст:
Чтобы ответы тренера были предсказуемыми, полезно вводить внутренние режимы поведения (не для пользователя, а для логики системы). Это повысит качество диалогов и снизит хаос в ответах.
Замечания / предложения:
Добавить внутренний флаг decision_mode, который выбирается по входным данным.
Расширить набор режимов до 5:
1) build — недостаточно данных, тренер достраивает план
2) suggest — план можно улучшить, тренер предлагает изменения
3) warn — план возможен, но рискован
4) refuse — план невозможен, тренер отказывает и предлагает альтернативы
5) ok — всё хорошо, план соответствует требованиям



Контекст/замечания/предложение для post‑MVP
Контекст
Хотим усилить “власть” тренера для жёсткого стиля и добавить автоматизацию изменений плана.
Замечания
Полная автоматическая перестройка плана без подтверждения снижает доверие и может раздражать пользователя. Это рисковано на MVP.
Предложение (post‑MVP)
Добавить режим условного автосогласия:
пользователь может включить “разрешаю тренеру перестраивать план в пределах N%”
или “перестраивать только дедлайны/нагрузку”
тогда жёсткий тренер сможет изменять план в рамках разрешения без ручного подтверждения


Post‑MVP: История смены тренеров + “handoff” контекст
Контекст:
После смены тренера полезно сохранять историю, чтобы учитывать прошлые предпочтения пользователя и улучшать адаптацию.
Предложение:
1) Хранить историю смен тренеров (кто, когда, на какой цели).
2) Передавать “handoff‑контекст” новому тренеру — краткое резюме поведения пользователя:
предпочитаемый темп
склонность выполнять/пропускать
реакция на строгий тон
уровень мотивации
Зачем:
Новый тренер не “начинает с нуля”
Повышается персонализация и стабильность стиля

Контекст:
Чтобы разнообразить уведомления, можно использовать AI‑перефразирование на основе уже подготовленных фраз.
Идея:
В профиле тренера хранится базовый набор фраз
AI генерирует новые сообщения “по образу”
Стиль и тон сохраняются, но тексты становятся разнообразнее
Плюсы:
больше вариативности
меньше “однообразных” уведомлений
легче масштабировать
Минусы:
риск отклонения от стиля
дополнительные запросы к LLM
сложнее контролировать качество
Вывод:
На этапе MVP используем только готовые фразы. AI‑перефразирование — после.

Post‑MVP: умный сдвиг и пересборка графика
Контекст:
Сейчас при опережении/отставании используется механический сдвиг дат. Это ограничивает гибкость.
Нужно реализовать:
1) AI‑сдвиг графика
AI предлагает новые даты с учётом контекста и ответа пользователя.
2) Механический сдвиг по правилам
Простая логика сдвига на N дней, если пользователь не хочет сложной перестройки.
3) Полная пересборка графика
Возможность пересобрать план при больших отставаниях или ускорении.

Post‑MVP: Виджет/уведомления с быстрым прогрессом
Контекст:
Пользователю удобно отмечать прогресс без открытия приложения.
Нужно реализовать:
Виджет на рабочий экран / интерактивные уведомления
Возможность отметить выполнение задачи прямо из уведомления
Быстрые действия: “Выполнил”, “Частично”, “Отложить”
